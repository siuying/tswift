# SwiftUI

SwiftUI is Apple's declarative UI framework. tswift renders it **without a Swift
toolchain, LLVM, or native compilation**: the interpreter instantiates a `View`,
runs `body`, produces a host-neutral **UIIR** view tree, diffs it into a **patch
stream**, and a dependency-free host applies those patches to real UI.

**Current status: foundational tiers + breadth shipped.** The Counter slice (v1)
plus Tiers 2тАУ5 тАФ composition, dynamic collections, input controls, and
observation тАФ are implemented and gated by deterministic goldens. On top of that,
the **C1тАУC7 breadth batches** widened the surface: universal text/styling
modifiers, layout arguments, structural containers (`Group`/`Divider`/
`ScrollView`), decoration (`border`/`shadow`/`clipShape`), content views
(`Label`/`Image`/`ProgressView`), lazy stacks & grids, `Form`, and control
styling тАФ each landed in all three appliers (runtime, web, iOS) with goldens.
The latest breadth pass widened it further: the full `Animation` curve family
(`.default`, `timingCurve`, `interpolatingSpring`, `interactiveSpring`) and
`AnyTransition` combinators, the system `Color` palette with `.opacity(_:)`,
Text typography (kerning/tracking/baseline/monospacing), graphic effects
(blur/brightness/contrast/saturation/grayscale/rotationEffect via `Angle`),
and list/scroll styling plus token-enum modifiers (blendMode, controlSize,
symbolRenderingMode, redacted, truncationMode). A further pass widened the
modifier surface again: value/nested-view modifiers (`position`, `mask`,
`contextMenu`, `listRowBackground`, `EdgeInsets`-backed `listRowInsets`), a
family of single-closure event handlers (`onHover`, `onOpenURL`, `refreshable`,
the edit/pencil/hover commands), and тАФ via a **contextual typed-token seam** тАФ
dozens of token modifiers whose leading-dot argument resolves against a
dedicated parameter type instead of by global uniqueness (`colorScheme`,
`symbolVariant`, `imageScale`, `keyboardType`, `dynamicTypeSize`,
`textContentType`, `scrollBounceBehavior`, and more). Modifier closures now
execute through the same event-to-patch path, including state-changing
gesture, lifecycle, refresh, submit, and change handlers. Closure-backed
`Binding` transforms, child `$state` projections, and keyed
`@Environment` propagation are also wired. Async (Tier 6) and
`GeometryReader` (Tier 7) are not started; navigation has a partial
single-window `NavigationStack`/`NavigationLink` slice.

**Legend:** тЬЕ Implemented ┬╖ ЁЯЯб Partial ┬╖ тмЬ Missing ┬╖ ЁЯЪл Out of scope.
The surface count below tracks individual API members across the iOS SDK
SwiftUI interface тАФ it is much lower than the tier roadmap suggests because
each shipped view exposes many initializers and modifiers we don't need yet.
The CLI, WASM, and FFI hosts accept one `App тЖТ Scene тЖТ WindowGroup` entry (via
`@main` or an explicit `MyApp.main()`), unwrap its root view, and drive the
ordinary render session. Window lifecycle, multiple windows, platform scenes,
and commands remain deliberately unsupported.
The **functional** coverage тАФ enough to render and interact with real view
hierarchies тАФ is further along, as the tiers below show. Generated by
`python3 tools/framework-inventory/coverage.py --framework swiftui --emit-json`.

The same patch protocol drives two hosts: the web `<swiftui-canvas>` custom
element, and a native iOS SwiftUI applier used for screenshot verification.

## Tier roadmap

| Tier | Mechanism | Status |
|------|-----------|--------|
| **v1** | Counter slice тАФ `VStack`/`Text`/`Button`, `@State`, modifiers, full renderтЖТdiffтЖТpatch loop | тЬЕ |
| **Tier 2** | View composition & demo tabs (Greeting/Stack/Profile) тАФ `HStack`/`ZStack`/`Spacer`, shapes, `Toggle` | тЬЕ |
| **Tier 3** | Dynamic collections & identity тАФ `List`/`Section`/`ForEach`, keyed diff with `move`/`insert`/`remove` | тЬЕ |
| **Tier 4** | Bindings & input controls тАФ `TextField`/`SecureField`/`Slider`/`Stepper`/`Picker` | тЬЕ |
| **Tier 5** | Observation, bindings & environment тАФ `ObservableObject`/`@Published`/`@StateObject`/`@EnvironmentObject`/`@Binding` | ЁЯЯб |
| **Tier 6** | `Task` & async push channel | тмЬ |
| **Tier 7** | Layout & geometry тАФ `GeometryReader`, alignment | тмЬ |
| **Tier 8** | Navigation & presentation тАФ single-window `NavigationStack`/`NavigationLink` slice | ЁЯЯб |

Tier 5 is partial: observation, child binding writeback, and keyed
`@Environment(\.keyPath)` propagation are implemented; collection bindings,
transaction/animation semantics, and broader observation coverage remain.

## Verification

Every slice is checked by an autonomous loop тАФ no Apple toolchain required to
gate a change.

| Layer | What it checks | Status |
|-------|----------------|--------|
| **A тАФ Surface** | API inventory vs the iOS SDK SwiftUI interface | тЬЕ |
| **B тАФ UIIR goldens** | initial render produces the exact expected view tree | тЬЕ |
| **C тАФ Interaction goldens** | events produce the exact expected patch stream | тЬЕ |
| **D тАФ Screenshot diff** | render UIIR to real SwiftUI (iOS) and the web host, screenshot both | ЁЯЯб |

**Layer D** runs both halves locally today: a native `UiirRenderer` SwiftPM
package builds real SwiftUI from the UIIR fixtures and snapshots them, and a
Playwright harness drives the `<swiftui-canvas>` element through the same
`mount тЖТ applyPatches` loop. Both capture a **device ├Ч appearance matrix**
(iPhone + iPad, light + dark), so token/CSS drift between the web and native
renderers is surfaced side by side. Perceptual-diff tooling and a CI job are the
remaining pieces.
