# SwiftUI

SwiftUI is Apple's declarative UI framework. tswift renders it **without a Swift
toolchain, LLVM, or native compilation**: the interpreter instantiates a `View`,
runs `body`, produces a host-neutral **UIIR** view tree, diffs it into a **patch
stream**, and a dependency-free host applies those patches to real UI.

**Current status: foundational tiers + breadth shipped.** The Counter slice (v1)
plus Tiers 2‚Äì5 ‚Äî composition, dynamic collections, input controls, and
observation ‚Äî are implemented and gated by deterministic goldens. On top of that,
the **C1‚ÄìC7 breadth batches** widened the surface: universal text/styling
modifiers, layout arguments, structural containers (`Group`/`Divider`/
`ScrollView`), decoration (`border`/`shadow`/`clipShape`), content views
(`Label`/`Image`/`ProgressView`), lazy stacks & grids, `Form`, and control
styling ‚Äî each landed in all three appliers (runtime, web, iOS) with goldens.
The latest breadth pass widened it further: the full `Animation` curve family
(`.default`, `timingCurve`, `interpolatingSpring`, `interactiveSpring`) and
`AnyTransition` combinators, the system `Color` palette with `.opacity(_:)`,
Text typography (kerning/tracking/baseline/monospacing), graphic effects
(blur/brightness/contrast/saturation/grayscale/rotationEffect via `Angle`),
and list/scroll styling plus token-enum modifiers (blendMode, controlSize,
symbolRenderingMode, redacted, truncationMode). A further pass widened the
modifier surface again: value/nested-view modifiers (`position`, `mask`,
`contextMenu`, `listRowBackground`, `EdgeInsets`-backed `listRowInsets`), a
family of single-closure event handlers (`onHover`, `onOpenURL`, `refreshable`,
the edit/pencil/hover commands), and ‚Äî via a **contextual typed-token seam** ‚Äî
dozens of token modifiers whose leading-dot argument resolves against a
dedicated parameter type instead of by global uniqueness (`colorScheme`,
`symbolVariant`, `imageScale`, `keyboardType`, `dynamicTypeSize`,
`textContentType`, `scrollBounceBehavior`, and more). Async (Tier 6),
`GeometryReader` (Tier 7), and navigation (Tier 8) are not started.

**Legend:** ‚úÖ Implemented ¬∑ üü° Partial ¬∑ ‚¨ú Missing ¬∑ üö´ Out of scope.
The surface count below tracks individual API members across the iOS SDK
SwiftUI interface ‚Äî it is much lower than the tier roadmap suggests because
each shipped view exposes many initializers and modifiers we don't need yet.
The host accepts one `App ‚Üí Scene ‚Üí WindowGroup` entry (via `@main` or an
explicit `MyApp.main()`), unwraps its root view, and drives the ordinary render
session. Window lifecycle, multiple windows, platform scenes, and commands
remain deliberately unsupported.
The **functional** coverage ‚Äî enough to render and interact with real view
hierarchies ‚Äî is further along, as the tiers below show. Generated by
`python3 tools/framework-inventory/coverage.py --framework swiftui --emit-json`.

The same patch protocol drives two hosts: the web `<swiftui-canvas>` custom
element, and a native iOS SwiftUI applier used for screenshot verification.

## Verification

Every slice is checked by an autonomous loop ‚Äî no Apple toolchain required to
gate a change.

| Layer | What it checks | Status |
|-------|----------------|--------|
| **A ‚Äî Surface** | API inventory vs the iOS SDK SwiftUI interface | ‚úÖ |
| **B ‚Äî UIIR goldens** | initial render produces the exact expected view tree | ‚úÖ |
| **C ‚Äî Interaction goldens** | events produce the exact expected patch stream | ‚úÖ |
| **D ‚Äî Screenshot diff** | render UIIR to real SwiftUI (iOS) and the web host, screenshot both | üü° |

**Layer D** runs both halves locally today: a native `UiirRenderer` SwiftPM
package builds real SwiftUI from the UIIR fixtures and snapshots them, and a
Playwright harness drives the `<swiftui-canvas>` element through the same
`mount ‚Üí applyPatches` loop. Both capture a **device √ó appearance matrix**
(iPhone + iPad, light + dark), so token/CSS drift between the web and native
renderers is surfaced side by side. Perceptual-diff tooling and a CI job are the
remaining pieces.

- SwiftUI `View`s built from the implemented views, controls, and shapes above

- Dynamic `List`/`ForEach` with add/remove/reorder
- A single-window `@main App` / explicit `MyApp.main()` entry, including
