# Swift Testing

[Swift Testing](https://github.com/swiftlang/swift-testing) is Apple's
attribute- and macro-based test framework (`import Testing`, `@Test`,
`#expect`, `#require`). tswift implements a native subset directly in the
interpreter ‚Äî no macro-expansion engine: `@Test`/`@Suite` are discovered
structurally from the typed AST, and `#expect`/`#require` are freestanding
macros registered the same way `#Predicate` is for SwiftData.

## Supported surface

| Status | Feature | Notes |
|--------|---------|-------|
| ‚úÖ | `@Test` free functions | Top-level `func` |
| ‚úÖ | `@Test` methods on `struct`/`class` | With or without an enclosing `@Suite` |
| ‚úÖ | Display names ‚Äî `@Test("‚Ä¶")` | First unlabeled string argument on the attribute |
| ‚úÖ | `#expect(_:)` | Soft failure: records an issue, test body continues |
| ‚úÖ | `#require(_:)` | Hard failure: aborts the test immediately |
| ‚úÖ | `try #require(_:)` optional unwrap | Unwraps and returns the value, or aborts |
| ‚úÖ | Failure detail | Expression source spelling + operand values for `==` `!=` `<` `>` `<=` `>=` and unary `!` |
| ‚úÖ | Throwing tests (`func f() throws`) | An uncaught `throw` is one failing issue |
| ‚úÖ | Async tests (`func f() async` / `async throws`) | Runs on the existing cooperative executor |
| ‚úÖ | `@Suite` + display name | `@Suite("‚Ä¶") struct MathSuite { ‚Ä¶ }` |
| ‚úÖ | Nested suites | Traits on an outer suite are inherited by nested suites/tests |
| ‚úÖ | `.disabled("reason")` | Skips the test; reason is printed |
| ‚úÖ | `.enabled(if: condition)` | Condition evaluated once at run start |
| ‚úÖ | `@Test(arguments:)` parameterized tests | One test case per element; multiple collections expand as a cartesian product; `zip(_:_:)` pairs element-wise |
| ‚úÖ | `Issue.record("‚Ä¶")` | Soft failure from inside test/helper code, without an `#expect` |
| üü° | Exit codes | `0` all pass (skips only); `1` on any failure, issue, or compile error. Discovering zero tests is **not** an error (exits `0` with an explicit "0 tests" summary) |
| ‚¨ú | `#expect(throws:)` / `throws: Never.self` | Closure-form throwing matchers |
| ‚¨ú | `.tags(‚Ä¶)` + tag-based filtering | Needs a `Tag` type and `--filter` tag syntax |
| ‚¨ú | `.bug(‚Ä¶)` | Report-only annotation |
| ‚¨ú | `.timeLimit(‚Ä¶)` | Needs a host timer policy for a hard kill |
| ‚¨ú | `confirmation { }` | Async confirmation helper |
| ‚¨ú | `withKnownIssue` | Expected-failure semantics |
| ‚¨ú | `CustomTestStringConvertible` | Custom value formatting in failure messages |
| ‚¨ú | Parallel test execution | tswift v1 runs **serially** (one interpreter instance); real Swift Testing parallelizes by default |

## Not yet supported (explicit non-goals for now)

- **XCTest** ‚Äî a separate framework; no dual-framework coexistence.
- **UI tests** (`XCUIApplication`) / **performance tests** (`XCTMetric`) ‚Äî tswift has no host/process model for these.
- **Full macro expansion** ‚Äî `@Test`/`@Suite`/`#expect`/`#require` are handled as
  structural discovery + two freestanding-macro builtins, not general
  compiler-plugin macro expansion.
- **`@testable` / true module isolation** ‚Äî multi-file programs are
  concatenated into one compilation unit; the system under test and its
  tests are treated as a single unit.
- **Compiler-plugin diagnostics** (e.g. "this expectation will always fail" warnings) ‚Äî would need type-aware constant folding in sema.
- **JSON / xUnit reporters** ‚Äî console output only.
- **Wasm/browser test running** ‚Äî `tswift test` is a CLI command; the site's
  live playground does not run test suites (see below).

## CLI usage

```text
tswift test <file.swift> [more.swift ...]
tswift test <dir>                              # all *.swift in the directory
tswift test <dir-with-Package.swift> [--target <testTargetName>]
tswift test ‚Ä¶ [--filter <substring>]
```

`--filter` matches (case-sensitively, substring) against a test's display id ‚Äî
its function name, `Suite/test` path, or display name string.

### Example

```swift
import Testing

@Test func addsUp() {
    #expect(1 + 1 == 2)
}

@Test("subtraction works")
func subtracts() {
    #expect(5 - 2 == 3)
}

@Test func broken() {
    #expect(2 + 2 == 5)
}

@Suite("Math Suite")
struct MathSuite {
    @Test func passes() throws {
        let maybe: Int? = 5
        let x = try #require(maybe)
        #expect(x == 5)
    }

    @Test(.disabled("flaky"))
    func skipped() {
        #expect(false)
    }
}

@Test(arguments: [1, 2, 3])
func isPositive(n: Int) {
    #expect(n > 0)
}
```

Filtering to just the suite:

```text
$ tswift test Basic.swift --filter Math
Test run started.
‚úî Test Math Suite/passes() passed after 0.000 seconds.
```

Running a SwiftPM-style package selects its `.testTarget`s:

```text
$ tswift test . --target AppTests
Test run started.
‚úî Test addWorks() passed after 0.000 seconds.
Test run with 1 test, 1 passed after 0.000 seconds.
```

With no `--target`, every declared `.testTarget` in `Package.swift` runs, one
after another (tswift v1 is serial ‚Äî see "Not yet supported" above).

### Exit codes

| Exit code | Meaning |
|---|---|
| `0` | All discovered tests passed (or were only skipped); this includes a run that discovers zero tests |
| `1` | Any test failed, any issue was recorded, or the sources failed to compile/analyze |

## How it renders

Discovery walks the typed AST after `analyze_program` ‚Äî no compiler plugin, no
symbol mangling, no binary reflection. `#expect`/`#require` are freestanding
macros registered on the interpreter, the same seam SwiftData uses for
`#Predicate`. Console output mirrors `swift test`'s structure and wording,
with ASCII pass/fail/skip markers (`‚úî` `‚úò` `‚Ü∑`) instead of SF Symbol
codepoints, so CI logs stay readable on any terminal.

Because the playground runs a single evaluated snippet rather than a
file/package tree with a `Testing` install + report loop, `tswift test` is
CLI-only for now; wiring a wasm test runner into the playground is tracked
separately and not part of this page's scope.
