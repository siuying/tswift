# Frontend Pipeline

The tswift frontend is a **pure-Rust Swift compiler front-end** that turns Swift
source into a stable, runtime-facing typed AST. There is no C code, no LLVM,
no `unsafe`, and no Swift toolchain involved.

---

## Stage 1 тАФ Lexer (`tswift-lexer`)

The lexer converts raw Swift source into a stream of **tokens**. It handles:

- All Swift literal forms: integers (dec/hex/oct/bin with `_` separators),
  floats (including hex exponents), booleans, `nil`, strings (including
  multiline `"""`, raw `#"тАж"#`, extended delimiters, and `\(expr)` interpolation).
- Unicode identifiers and operator characters.
- Line comments (`//`), block comments (`/* тАж */`), and nested block comments.
- Regex literals `/тАж/` and `#/тАж/#`.

String interpolation is particularly tricky: the lexer emits a token stream
that the parser then re-enters for the interpolated expression, allowing
arbitrary nesting.

---

## Stage 2 тАФ AST (`tswift-ast`)

The AST crate defines the **node types** shared by all pipeline stages. Every
node carries:

- `kind: NodeKind` тАФ the syntactic kind (e.g. `FuncDecl`, `BinaryExpr`, `CallExpr`)
- `text: &str` тАФ the source text span
- `line: u32` тАФ source line number

After semantic analysis, nodes also carry:

- `resolved_type: Option<SwiftType>` тАФ the inferred Swift type
- `modifiers: Modifiers` тАФ decoded declaration modifiers (`mutating`, `static`, тАж)

---

## Stage 3 тАФ Parser (`tswift-parser`)

A **recursive-descent** parser that consumes the token stream and produces an
untyped AST. Highlights:

- Full expression hierarchy (primary тЖТ postfix тЖТ unary тЖТ binary with precedence
  climbing)
- Pattern matching (`switch`, `if case`, `for case`, destructuring)
- All declaration forms: `func`, `var`/`let`, `class`, `struct`, `enum`,
  `protocol`, `extension`, `actor`, `typealias`, `operator`, `precedencegroup`
- Generics: `<T: P>`, `where` clauses, parameter packs
- Attributes: `@escaping`, `@autoclosure`, `@discardableResult`, `@main`, тАж
- Concurrency: `async`, `await`, `actor`, `@MainActor`, `async let`

## Stage 4 тАФ Semantic Analysis (`tswift-sema`)

The semantic analysis pass resolves **types** and **names**:

- **Type inference** тАФ bidirectional propagation; literal types from context
- **Name resolution** тАФ lexical scoping, module-level declarations, cross-file
  references
- **Overload resolution** тАФ function overloads, operator overloads, method dispatch
- **Protocol conformance** тАФ builds a witness table for each `struct`/`class`/`enum`
  conformance declaration
- **Associated type** binding тАФ resolves `associatedtype` placeholders to concrete types

The output is a **typed AST** where each expression node carries its resolved
`SwiftType`.

---

## Stage 5 тАФ Compat Lowerer (`tswift-frontend`)

The compat lowerer is the bridge between the typed frontend AST and the
**runtime-facing AST** (`Analysis` / `Node` / `NodeKind`). Its job:

1. Drive the full pipeline (lexer тЖТ parser тЖТ sema) for one or more source files.
2. Lower the typed frontend AST into the stable `NodeKind` enum the runtime
   has always depended on.
3. Expose a single `Analysis::analyze(sources)` entry point.

This decoupling means the runtime never depends on frontend internals тАФ it only
sees `NodeKind` variants and `SwiftType` tags. Upgrading the frontend (e.g. adding
a new parse rule) does not require touching the runtime.

---

## Why no `unsafe`?

The entire frontend stack is `#![forbid(unsafe_code)]`. Rust's ownership
model gives us memory safety during parsing for free тАФ no garbage collector,
no arena lifetime tricks, just owned `Vec` and `Box` trees.
